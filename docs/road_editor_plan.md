# Road Editor Plan

道路エディタの今後の設計方針をまとめたメモです。  
現状の編集モードと `road_data_format.json` の構造を前提に、道路ネットワークを快適にオーサリングするための考え方を整理します。

## 参照実装

v2 の viewer state では、以下の 5 モードを切り替えて編集している。

- `SELECT`
  ポリラインの選択、追加、削除、結合
- `EDIT`
  道路制御点の追加、移動、削除、分割
- `VERTICAL_CURVE`
  縦断曲線ポイントの追加、移動、削除
- `BANK_ANGLE`
  バンク角ポイントの追加、移動、削除
- `LANE_SECTION`
  車線セクションポイントの追加、移動、削除

また実装上は、道路上の各種ポイント編集が `u` 値ベースで統一されている。

- `MODE_PARM_MAP` でモードごとの Multiparm 名と `u` パラメータ名を切り替える
- HUD 表示もモードごとに切り替える
- ポイント追加、選択、削除、ドラッグの基本操作は各モードで共通化されている

このため、新しい `Metadata` モードも、v2 の仕組みを踏襲して追加するのが自然。

## 目的

- 道路ネットワークを気持ちよく素早く作れること
- 後から交差点まわりを細かく調整できること
- 停止線や進入位置のような局所情報を破綻なく持てること
- 今ある道路エディタの操作思想をできるだけ崩さないこと

## 基本方針

道路ツール全体としては、`道路主導` を基本にするのが良い。

- 普段のオーサリングは道路を引くことが中心
- 交差点は道路の接続結果として生成または更新する
- 必要なときだけ交差点まわりを局所的に調整する

この方針にすると、広い道路網を作る操作が軽くなり、ユーザーが毎回交差点を先に作らなくて済む。

## なぜ道路主導がよいか

- ユーザーはまず道路の流れや接続を描きたいことが多い
- 交差点主導にすると、大きなネットワークの初期構築が重くなりやすい
- 現在のエディタも道路に対するポイント編集が中心なので、既存の思想を活かしやすい

ただし、交差点を完全に自動化するのではなく、交差点には交差点としての責務を残す。

- 道路はネットワークの骨格を定義する
- 交差点は接続関係と接続部の形状ルールを持つ
- 停止線や進入位置は、道路が交差点へ入る側の属性として持つ

## 現在の編集モード

現在の道路エディタは、道路に対して以下を切り替えて編集している。

- 道路頂点
  クロソイドカーブの制御点
- 縦断曲線
  ポイント配置
- バンク角
  ポイント配置
- 車線数
  ポイント配置

この構成は、「道路に沿って何かをポイントで定義する」という操作体系で統一されている。

v2 の実装を見る限り、この整理はデータ仕様だけでなく viewer state の操作モデルにも反映されている。

- 線形本体は直接ポイントを編集する
- 縦断、バンク角、車線は道路上の `u` 位置に対してポイントを置く
- モードを切り替えて、同じ道路に対して異なるレイヤーを編集する

したがって、停止線や進入位置を扱う場合も、この「道路に対する別レイヤーをモード切替で編集する」思想に合わせるのが良い。

## 追加したいモード

次の候補として `メタデータポイント配置モード` を追加するのが自然。

### 狙い

- 停止線
- 進入位置
- 横断歩道
- 制限速度変更
- 信号や進入制御

といった道路付随情報を、既存のポイント編集と近い操作感で扱えるようにする。

### ただし内部設計では分ける

UI 上は 1 つの `Metadata` モードでもよいが、内部では次の 2 種類に分けたほうがよい。

- 一般メタデータ
  道路上の任意位置に置けるもの
- 端点メタデータ
  道路の始点または終点に結びつくもの

この分離はかなり重要。

- 停止線は道路全体の連続属性ではなく、交差点進入端の属性
- 進入位置も道路端に強く結びつく
- 制限速度などは道路上の任意位置で変化させたい

そのため、見た目は同じ「ポイント配置」でも、意味は分けて持つべき。

特に v2 との整合を考えると、次のようにするのが実装しやすい。

- `Metadata` を 1 つの新モードとして追加する
- ただし内部では `端点系` と `一般系` をサブタイプで分ける
- `u` 値で管理できるものは既存方式を使う
- 始点または終点に拘束されるものは、`start/end` を明示して扱う

## 停止線と進入位置の考え方

停止線や進入位置は、`交差点そのものの共有属性` ではなく、`道路の端が交差点に入るときの属性` として扱うのが自然。

つまり、以下のような整理になる。

- 交差点
  接続関係、接続順、交差点領域、角の処理などを持つ
- 道路端
  進入位置、停止線位置、横断歩道、信号制御などを持つ

この設計にすると、同じ交差点でも接続道路ごとに異なる停止線位置を持てる。

## データ設計の考え方

現状の `roads` に対して、端点ごとのメタデータブロックを追加する形が有力。

例:

```json
{
  "id": "road_01",
  "startIntersectionId": "intersection_a",
  "endIntersectionId": "intersection_b",
  "endMetadata": {
    "start": {
      "entryOffset": 0.0,
      "stopLineOffset": -2.0
    },
    "end": {
      "entryOffset": 1.0,
      "stopLineOffset": 0.5
    }
  },
  "metadataPoints": [
    {
      "type": "speed",
      "s": 120.0,
      "value": 50.0
    }
  ]
}
```

### 端点メタデータ候補

- `entryOffset`
  交差点接続点からの前後オフセット
- `stopLineOffset`
  停止線位置の前後オフセット
- `crosswalkOffset`
  横断歩道位置の前後オフセット
- `signalId`
  信号制御との関連付け
- `yieldType`
  一時停止、停止線なし、優先など

### 一般メタデータ候補

- 制限速度
- 路面種別
- 白線種別
- 追い越し可否
- 路肩種別

## オフセットの基準

停止線や進入位置は、道路中心線に沿った距離オフセットで持つのが扱いやすい。

- 基準点は道路端の接続位置
- 正負で前後を表す
- 交差点内に停止線が入るケースも表現できる

例:

- `entryOffset = 0.0`
  既定の接続位置
- `stopLineOffset = -2.0`
  進入基準点より交差点内へ 2m
- `stopLineOffset = 3.0`
  進入基準点より道路側へ 3m

## エディタ UI の方向性

今の操作系を活かすなら、まずは `Metadata` モードを 1 つ追加するのが分かりやすい。

### 想定 UI

- モード切り替えに `Metadata` を追加
- メタデータ種別を選ぶ
  `StopLine`, `Approach`, `Speed`, `Crosswalk` など
- 種別によって配置ルールを変える

### 配置ルール

- `StopLine` や `Approach`
  始点または終点近傍にのみ配置可能
- `Speed`
  道路上の任意位置に配置可能

### 表示したいギズモ

- 交差点境界
- 進入位置ハンドル
- 停止線ハンドル
- 横断歩道ハンドル

最初から専用モードを増やしすぎず、まずは `Metadata` モードの中で端点系メタデータを扱えるようにするとよい。

v2 の HUD 構成に寄せるなら、追加モードは次のイメージになる。

- モード名
  `Metadata Editor`
- 基本操作
  `LMB` で追加または選択、`Del` で削除、`f` でフォーカス
- 必要ならサブタイプ切替
  `StopLine`, `Approach`, `Speed`, `Crosswalk`

つまり、ユーザー体験としては既存の `Vertical Curve Editor` や `Lane Editor` と同列に見せるのが良い。

## 道路と交差点の責務分担

### 接続トポロジーとは

ここでいう `接続トポロジー` は、交差点の論理的な接続構造を指す。

- 何本の道路が接続しているか
- どの道路が接続しているか
- それぞれが道路の始点なのか終点なのか
- 接続順がどう並ぶか
- T字、十字、多差路のどれに近いか

これは「何がどうつながっているか」の情報であり、停止線位置や進入位置のような局所詳細は含めない。

### 交差点形状ルールとは

ここでいう `交差点形状ルール` は、接続された道路群から交差点形状をどう生成するかの規則を指す。

- 通常交差点か
- ラウンドアバウトか
- 交差点内部の生成方式
- 進入部の既定カット長
- コーナーR
- ラウンドアバウトの外径
- アイランド半径

つまり、`接続トポロジー` が論理構造、`交差点形状ルール` が見た目や生成方式の規則である。

### 道路が持つもの

- 平面線形
- 縦断
- バンク角
- 車線数や車線構成
- 幅員
- 一般メタデータ
- 端点メタデータ
- 接続先交差点 ID

### 交差点が持つもの

- 接続道路の集合
- 接続道路の始点/終点の別
- 接続順や接続角度
- 交差点タイプ
- 生成アルゴリズム種別
- 進入距離の既定値
- 交差点領域の生成ルール
- 交差点内部形状
- ラウンドアバウト用の半径や島形状パラメータ

### 交差点が持たないほうがよいもの

- 各道路ごとの停止線位置
- 各道路ごとの進入位置微調整
- 各道路ごとの横断歩道位置
- 各道路ごとの信号制御詳細
- 各道路ごとの車線構成そのもの
- 各道路ごとの端点メタデータの最終値

これらは道路端ごとの差が大きく、道路側が所有したほうが将来の変更に強い。

### なぜ交差点側で持ちすぎないほうがよいか

IntersectionEditor-v2 のように、交差点が接続道路群を見て後段生成される構造自体は問題ない。
ただし、交差点側が道路端の詳細値まで所有し始めると、道路構成変更への追従が難しくなる。

- 道路分割時に値の引き継ぎ先が曖昧になる
- 道路の始点/終点の付け替え時に整合が崩れやすい
- 車線数変更時に交差点側の保持値が古くなりやすい
- 停止線や進入位置の編集責務が道路と交差点で二重化しやすい

そのため、交差点は道路を `読む側` にとどめるのがよい。

- 道路が端点メタデータを持つ
- 交差点はそれを参照して形状生成する
- 交差点側ではコピーを正として持たない

この設計にすると、道路構成が変わっても、編集責務の所在がぶれにくい。

### 境界にあるもの

以下は交差点にも見えるが、実際には道路端ごとの差が大きいので道路側に置くほうが良い。

- 停止線
- 横断歩道の基準位置
- 信号や進入制御
- 優先関係の局所フラグ

## 最初の実装ステップ

段階的に進めるのが安全。

### Phase 1

- `Metadata` モードを追加
- 端点メタデータとして `entryOffset` と `stopLineOffset` だけ扱う
- 始点と終点にハンドルを表示する
- データ保存のみ対応し、生成ロジックは最小限
- v2 の `MODE_PARM_MAP` と HUD 切替の延長で実装する

### Phase 2

- 停止線のビジュアル表示
- 停止線位置のドラッグ編集
- 交差点境界との見た目の関係を表示
- JSON フォーマットへ保存

### Phase 3

- 横断歩道や信号などの端点メタデータ拡張
- 制限速度などの一般メタデータポイント追加
- 交差点内部生成と連動

## 迷いどころ

今後決める必要がある論点もある。

- `entryOffset` の基準をどこに置くか
- 停止線位置を `entryOffset` 基準にするか、道路端基準にするか
- 交差点境界を明示的に持つか、接続道路から暗黙計算するか
- メタデータポイントを 1 配列にまとめるか、端点用と一般用で分けるか
- 将来的なレーン単位停止線に対応するか

## 現時点のおすすめ結論

現段階では、以下の方針が最もバランスが良い。

- オーサリングは道路主導
- 交差点は接続結果として生成し、必要時に調整
- エディタには `Metadata` モードを追加
- ただし内部データは `一般メタデータ` と `端点メタデータ` に分ける
- 停止線と進入位置は道路端メタデータとして持つ

この方針なら、今の道路エディタの操作感を壊さずに、交差点まわりの表現力を段階的に増やしていける。
