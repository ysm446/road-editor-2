# Standalone Road Editor Notes

## Goal

Cities: Skylines 2 のように道路を引けて、結果として路面、白線、歩道、ガードレール、標識などが追従するスタンドアローンの道路エディタを目指す。

単なるメッシュ編集ツールではなく、プロシージャルに再生成される道路ネットワークエディタとして設計する。

## これまでの試作
水平曲線や縦断曲線、バンク角、車線構成などを盛り込んだプロジェクト
https://github.com/ysm446/road-editor

## Core Direction

- Houdini らしいプロシージャル性を持たせる
- 線形を動かすと下流の結果が再計算される
- 道路を最初からポリゴンとして持たず、中間表現を主にする
- 交差点、白線、付属物はルール駆動で生成する
- GPU を最初から意識した描画基盤にする
- UI は DCC ツールのような構成にする

## Recommended Stack

現時点の本命案:

- UI: `Qt 6`
- 言語: `C++`
- 3D Viewport: `OpenGL` から開始
- データモデル: 独自の道路ネットワーク IR
- Python: 補助スクリプト、試作、将来のツール拡張用

理由:

- `Qt` は DCC 風のパネル、ドック、ツリー、アイコン付きボタンに向いている
- `OpenGL` は `Vulkan` より実装コストが低く、初期のツール開発に向いている
- GPU 表示は最初から欲しいが、生成ロジックは最初は CPU 中心でも進められる
- 道路ジオメトリや付属物の表示負荷は高いため、描画は最初から GPU 前提にする

## Why Not Python Only

Python だけでも試作は可能だが、本格的な道路エディタの本命としては弱い。

理由:

- 大規模メッシュ更新
- GPU バッファ管理
- ピッキング
- インスタンシング
- 大量道路ネットワークの再計算

といった処理は最終的に `C++` のほうが安定しやすい。

そのため、実務的には

- `Python` 単独で完結させるより
- `C++` を中心に据え
- `Python` は補助や試作に回す

構成がよい。

## UI Direction

`ImGui` は主 UI にしない。

理由:

- 見た目が開発ツール寄りになりやすい
- 製品感のある DCC 風 UI に寄せにくい

UI は `Qt` を主にし、必要であれば `ImGui` は内部デバッグ表示のみに限定する。

想定する画面構成:

- 中央: 3D ビューポート
- 左: ツールパネル
- 右: パラメータパネル
- 下: ログ、タイムライン、情報表示
- 別タブ: ノードグラフ
- 別パネル: アウトライナ、属性ビュー

## Rendering Direction

### OpenGL vs Vulkan

初期開発では `OpenGL` を推奨。

`OpenGL` の利点:

- 導入しやすい
- `Qt` と相性がよい
- DCC 風ビューポートを作りやすい
- ギズモ、補助線、ピッキング、ハイライトを実装しやすい

`Vulkan` の利点:

- 高性能
- より低レベルに GPU を制御できる
- 将来的な拡張性が高い

ただし `Vulkan` は初期開発コストが高いため、最初から採用するとエディタ本体よりレンダラ開発に時間を取られやすい。

結論:

- まずは `OpenGL`
- ただしレンダラ層は抽象化しておき、将来的に差し替え可能にする

## Procedural Data Model

道路はメッシュではなく、まず次のような意味のある中間表現を持つ。

- 平面線形
- 縦断線形
- 横断構成
- レーン構成
- 接続関係
- 交差点情報
- station / offset ベースの属性

これを元に各生成器が結果を作る。

例:

- Road Surface Generator
- Lane Marking Generator
- Sidewalk Generator
- Guardrail Generator
- Sign Generator
- Intersection Generator
- Props Scatter Generator

## Houdini-like Procedural Behavior

求めている性質は以下。

- 線形を動かす
- 結果メッシュが再生成される
- 白線や付属物が追従する
- 交差点が自動で組み変わる

このためには、直接メッシュ編集をするのではなく、ノード的な依存関係で結果を更新する設計が向いている。

内部構成のイメージ:

- Alignment Node
- Profile Node
- Lane Config Node
- Surface Node
- Marking Node
- Intersection Node
- Props Node

## OCCT Note

`OCCT` は `Open CASCADE Technology` の略で、CAD 系の幾何カーネル。

できること:

- 曲線、サーフェス、ソリッドの生成
- 幾何演算
- B-Rep 管理
- STEP / IGES などの CAD データ入出力

将来的に精度重視の設計ツールへ寄せるなら有力だが、最初の道路エディタでは必須ではない。

まずは独自の道路向けジオメトリ生成で進め、必要になったら検討するのが現実的。

## MVP Proposal

最初に作る範囲は次の順がよい。

1. 地面上に道路中心線を引ける
2. 幅員とレーン数を変えられる
3. 接続時に自動で交差点を作る
4. 白線と歩道が追従する
5. 勾配を持てる
6. 付属物をルールで配置できる

## Architecture Summary

現時点での推奨アーキテクチャ:

- `Qt 6` ベースの DCC 風 UI
- `OpenGL` ベースの GPU ビューポート
- `C++` 中心の道路生成コア
- プロシージャルな道路ネットワーク IR
- Python は補助的に利用
